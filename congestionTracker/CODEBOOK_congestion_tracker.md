# Codebook – City Congestion Tracker

This codebook describes the **data files**, **database tables**, and key **variables** used in the City Congestion Tracker midterm project.

---

## 1. Synthetic data files

### 1.1 `intersections.csv`

Created by `generate_synthetic_congestion.py`.

- **name**  
  - Type: text  
  - Example: `"Intersection 4"`  
  - Description: Human‑readable label for the intersection (road junction).

- **lat**  
  - Type: numeric (double precision)  
  - Example: `40.0123`  
  - Description: Latitude coordinate for the intersection (synthetic around a central city point).

- **lon**  
  - Type: numeric (double precision)  
  - Example: `-73.9934`  
  - Description: Longitude coordinate for the intersection (synthetic around a central city point).

- **priority_level**  
  - Type: text  
  - Allowed values: `"normal"`, `"hospital_route"`, `"school_zone"`  
  - Description: Simple tag that marks whether the intersection is part of a critical corridor for emergency services.

### 1.2 `congestion_readings_raw.csv`

Created by `generate_synthetic_congestion.py`.

- **intersection_index**  
  - Type: integer  
  - Example: `3`  
  - Description: Zero‑based index that ties this reading back to a row in `intersections.csv`. This is mapped to the real `intersection_id` when loading into Supabase.

- **timestamp_utc**  
  - Type: text (ISO 8601 UTC timestamp)  
  - Example: `"2026-03-07T14:35:00+00:00"`  
  - Description: Time of the congestion reading, rounded to a 5‑minute interval in UTC.

- **congestion_level**  
  - Type: integer, range 0–100  
  - Example: `78`  
  - Description: Synthetic congestion score:
    - `0` = free flow  
    - `40–60` = moderate delay  
    - `80–100` = severe congestion / likely gridlock.

- **speed_kmh**  
  - Type: numeric  
  - Example: `18.5`  
  - Description: Approximate speed in km/h implied by the congestion level (lower speed = more congestion).

- **delay_sec**  
  - Type: numeric  
  - Example: `92.0`  
  - Description: Approximate additional delay in seconds associated with this reading.

- **source**  
  - Type: text  
  - Example: `"synthetic"`  
  - Description: Marks that this dataset is simulated rather than from a real sensor feed.

## 2. Supabase tables

The synthetic data is loaded into a Supabase PostgreSQL database with two main tables.

### 2.1 `intersections`

Columns:

- **id**  
  - Type: `uuid` (primary key)  
  - Description: Unique identifier for each intersection, generated by Supabase.

- **name**  
  - Type: `text`  
  - Description: Human‑readable intersection label (from `intersections.csv`).

- **lat**  
  - Type: `double precision`  
  - Description: Intersection latitude (synthetic).

- **lon**  
  - Type: `double precision`  
  - Description: Intersection longitude (synthetic).

- **priority_level**  
  - Type: `text`  
  - Description: Priority tag (`normal`, `hospital_route`, or `school_zone`).

- **created_at**  
  - Type: `timestamptz`  
  - Description: Timestamp when the row was inserted, default `now()` in Supabase.

### 2.2 `congestion_readings`

Columns:

- **id**  
  - Type: `bigserial` (primary key)  
  - Description: Unique identifier for each congestion reading row.

- **intersection_id**  
  - Type: `uuid`, foreign key → `intersections.id`  
  - Description: Which intersection this reading belongs to.

- **timestamp_utc**  
  - Type: `timestamptz`  
  - Description: UTC timestamp of the reading, rounded to a 5‑minute bucket.

- **congestion_level**  
  - Type: `integer`, range 0–100  
  - Description: Congestion score at that time and location.

- **speed_kmh**  
  - Type: `double precision`  
  - Description: Approximate speed in km/h at that intersection at that time.

- **delay_sec**  
  - Type: `double precision`  
  - Description: Approximate delay in seconds at that intersection at that time.

- **source**  
  - Type: `text`  
  - Description: Data source label (here set to `"synthetic"`).

Indexes:

- `idx_congestion_intersection_time` on `(intersection_id, timestamp_utc)` for fast time‑window queries per intersection.
- `idx_congestion_time` on `(timestamp_utc)` for fast queries over recent readings.

## 3. Key scripts and how they use the data

- **`generate_synthetic_congestion.py`**  
  - Generates:
    - `intersections.csv` (intersection metadata).
    - `congestion_readings_raw.csv` (time‑series congestion data).
  - Encodes realistic patterns (rush hours, weekday/weekend differences, random spikes).

- **`load_congestion_readings.py`**  
  - Fetches `intersections` from Supabase to map `intersection_index` → real `intersection_id`.  
  - Loads `congestion_readings_raw.csv` and inserts rows into `congestion_readings`.

- **`api_main.py`**  
  - Uses the `intersections` and `congestion_readings` tables to serve:
    - Live top‑congested intersections.
    - Time‑series per intersection.
    - Aggregated stats for AI summaries.

---

## 4. Interpretation notes

- The data is **synthetic** but designed to mimic real city traffic:
  - Morning and evening peaks around typical commute hours.
  - Lower congestion on weekends.
  - Random spikes to simulate incidents.
- Congestion scores are **unitless** indices (0–100) but are mapped to speed and delay to make examples more intuitive.
- Priority levels can be used to:
  - Filter the map to focus on hospital routes or school zones.
  - Highlight critical intersections in AI summaries.
